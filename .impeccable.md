# AutoTrack 360° — Design Context

## Users

### Propietario (Owner)
- Persona que tiene uno o más vehículos y quiere mantener un historial verificable de mantenimiento
- Contexto: usa la app desde el celular para registrar services, reparaciones, VTV
- Job to be done: tener pruebas confiables del estado de su vehículo para vender mejor o mantenerlo al día

### Taller (Workshop)
- Mecánico o taller de servicios que gestiona vehículos de clientes
- Contexto: usa la app desde el taller (mobile o desktop) para registrar trabajos con evidencia
- Job to be done: generar confianza con sus clientes, llevar un CRM acotado, diferenciarse de la competencia

### Comprador (Buyer)
- Persona evaluando un vehículo usado para compra
- Contexto: escanea un QR o recibe un link del vendedor
- Job to be done: verificar que el historial del vehículo es real y confiable antes de comprar

### Organizaciones B2B (Concesionarias, Aseguradoras, Financieras)
- Empresas que consultan historiales vehiculares via API
- Contexto: integración sistema a sistema, no usan la UI
- Job to be done: evaluar el estado de un vehículo para tomar decisiones de negocio

## Brand Personality

**Moderno — Deportivo — Profesional**

- **Voz**: directa, técnica pero accesible, sin jerga innecesaria
- **Tono**: seguro, conciso, orientado al amante de vehículos
- **Emoción principal**: **claridad y control** — el usuario siente que tiene todo organizado y bajo control
- **Identidad**: una herramienta que un entusiasta de autos confiaría y recomendaría

## Aesthetic Direction

### Visual Tone
- **Dark theme premium**: fondo slate profundo (#0f172a), superficies elevadas con gradientes sutiles
- **Accent cálido**: amber/naranja (#f59e0b) como color de acción y marca — evoca energía, confianza, automotriz
- **Tipografía limpia**: Inter para UI, JetBrains Mono para datos técnicos (patentes, hashes, IDs)
- **Elevación por gradiente**: las cards usan gradientes sutiles en lugar de sombras fuertes, con inner highlights casi imperceptibles

### Patterns
- Bordes sutiles (slate-700 a 50% opacidad) para separar sin pesar
- Rounded-xl (12px) en todos los componentes interactivos
- Animaciones sutiles: scale(1.02) en hover, transiciones de 200ms
- Mobile-first: bottom nav en mobile, header expandido en desktop
- Glow effects para CTAs importantes (sombra naranja difusa)

### Theme
- Solo dark mode (no hay light mode)
- Colores semánticos: verde = trust alto, amber = trust medio, rojo = trust bajo/peligro
- Badges y estados con fondo al 20% de opacidad + borde al 20% del color

### Anti-references
- NO debe verse como una app gubernamental o burocrática
- NO debe verse sobrecargado con datos — priorizar jerarquía visual clara
- NO colores fríos puros — el amber le da calidez al dark theme

---

## Design Principles

1. **Confianza visual**: cada dato verificado debe comunicar su nivel de confianza. Usar badges, iconos de verificación, y el scoring engine visualmente.

2. **Jerarquía clara**: en una pantalla con muchos datos (eventos, evidencias, scores), el usuario debe poder escanear rápidamente qué es importante. Títulos grandes, metadata secundaria en slate-400.

3. **Mobile-first, siempre**: el 80% del uso será desde el celular en el taller o en la calle. Botones grandes, scroll vertical, sin hover-only interactions.

4. **Feedback inmediato**: toda acción debe tener confirmación visual — toasts de éxito, spinners en botones, estados de carga skeleton.

5. **Datos, no decoración**: la interfaz debe ser funcional primero. Cada elemento visual debe servir a la comprensión del dato, no a la estética vacía.
